## High-level summary

- HDMI Type-A female SMD receptacle ("USB连接器 [HDMI型母座]" = "USB-family connector [HDMI-type female socket]" — SOFNG catalogs it alongside their USB connector line), right-angle mount, gold-plated contacts, single-orientation/keyed for small-device mounting, reel-tape packaging, RoHS compliant `[VERIFIED — datasheet p.1]`. 19 signal positions ("19P" per BOM) + 4 mechanical shield/ground tabs = 23 total pins, matching the netlist's `U8-1` through `U8-23` `[VERIFIED — datasheet p.3 footprint drawing, cross-checked against Cradle.NET]`.
- Ordered/approved as **Grade B** ("HDMI-519S [B级标]") per the datasheet's own sample-approval sheet `[VERIFIED — datasheet p.2]` — this selects the "B级" column throughout the p.4 reliability table (contact resistance, life cycles, temperature/humidity ratings), not the A or C columns.
- **This datasheet has zero pin-function content** — only mechanical dimensions (p.3), a photo, and reliability/soldering-process specs (p.4-5), same class of gap as TF-01A's scanned-image datasheet, just a different failure mode (a generic connector-house catalog sheet that assumes the reader already knows the standard HDMI Type-A signal assignment, rather than a broken PDF). `datasheet_index.py` correctly found nothing portable (0 pin/spec tables) — this isn't a tool miss like the EMMC card's, it's a genuinely contentless-for-pinout source document.
- **The pin table below is therefore drawn from the public HDMI Type-A standard pinout (this is standardized by the HDMI spec itself, not vendor-specific), not from this datasheet** `[UNVERIFIED — agent's general knowledge of the HDMI Type-A standard, not sourced from any document in this repo]`. Partially cross-checked: RK3576's own ballout independently confirms `HDMI_TX_D0P`/`D0N` through `D3P`/`D3N` (4 differential pairs), `HDMI_TX_CEC`, `HDMI_TX_SCL`/`SDA`, `HDMI_TX_HPDIN` all exist as real signals `[VERIFIED — RK3576 datasheet V1.6.tables.md, HDMI_TX_* pinmux rows]` — which confirms the *signal set* but not this specific connector's *pin numbering*. Given a TMDS polarity swap is a real, hard-to-debug failure mode, treat the pin table below as needing a second-source check (a known-good HDMI reference design) before finalizing layout — see Known gaps.
- **RK3576 appears to use a combined HDMI/eDP PHY**: the ballout shows shared pins (`HDMI_TX_D0P/EDP_TX_D0P`, etc.) with 4 differential pairs (`D0`-`D3`) rather than the classic "3 TMDS data pairs + 1 separate clock pair" naming. Exact `D0`-`D3` → TMDS Data0/Data1/Data2/Clock mapping not confirmed — no HDMI chapter found in the RK3576 TRM Part1 volume present in this repo (likely documented in a Part2 not on disk here). Also needs its own dedicated analog supplies (`HDMI_TX_EDP_TX_AVDDIO1V8`, `AVDDD0V75`, `AVDDCMN1V8`, `AVDDC0V75`) and a precision bias resistor (`HDMI_TX_REXT`) `[VERIFIED — RK3576 datasheet V1.6.tables.md]` — not this connector's concern directly, but necessary context for whoever wires the RK3576/Compute-SoC side of this net.
- **Pin table corrected 2026-07-21 — this connector's real physical pin numbering does NOT match the official HDMI Type-A signal-position numbering** `[MANUAL — user, 2026-07-21, verified via own implementation + a Digikey reference]`. The official spec's "pin 1 = Data2+" convention describes signal *position*, not necessarily this vendor's *physical pin numbering* — SOFNG numbers its physical pins shield-then-plus-then-minus per triad (1=Data2 Shield, 2=Data2+, 3=Data2-, ...) rather than plus-then-shield-then-minus. This is exactly the kind of discrepancy flagged as a risk in the prior session (a second-source check was recommended precisely because a polarity/position swap on a connector-house datasheet with no pin table is a real, hard-to-debug failure mode) — confirmed real here. See Pin table for the corrected mapping.
- **Pin `Type` corrected**: the TMDS differential pairs are `I` (inputs to this connector part), not `O`. Matches the same convention used on `AP6275S.md`/`EMMC04G-M627-Y02U.md`: `Type` describes the signal direction *relative to the part the card describes*, not relative to the overall board. RK3576 (not the connector) is the actual driver/source of the TMDS signal — the connector receives it from RK3576 and passes it through externally, so from the connector's own perspective it's an input, the same way `WL_REG_ON` is an "I" on `AP6275S.md` because the host drives it into the module.
- **Storage sheet's `EMMC_DS` FET is now built and confirmed in the netlist** (`Q2`, gate on `EMMC_DS_TOGGLE` — still a single-pin net, i.e. the RK3576 GPIO side isn't connected yet; drain on `NetQ2_2` with `R61`; source on `GND`) — matches what was described, unrelated to this connector.
- **HDMI's own level-shifting circuit is now implemented for `CEC`/`SCL`/`SDA`/`HPD`** (see Pin table for the per-signal detail) — `CEC`/`SCL`/`SDA` use a bidirectional single-MOSFET-per-line shifter (`Q3`/`Q4`/`Q5`, `LBSS138LT1G`), gates biased from a new `INTERFACES_3V3` net, low-side pull-ups (new 4.7kOhm `TFCR0603-10W-C-4701FT-1K` resistors, `R62`-`R69` — `R62` added to `standard-parts.md` as the staged instance of this newly-standardized value) referenced to `INTERFACES_3V3`, high-side pull-ups referenced to a new `INTERFACES_5V` net (also feeding the connector's own `+5V` pin — efficient reuse of one local rail for both jobs). `INTERFACES_3V3` being the gate-bias/low-side rail is itself useful evidence: it confirms RK3576's `HDMI_TX_CEC`/`SCL`/`SDA`/`HPDIN` were designed against a **3.3V** GPIO bank, resolving the earlier open question about which VCCIO domain hosts them.
- **`HPD` (`Q6`) intentionally uses a simpler, different (correct) circuit** — confirmed by the user: since `HPD` only flows one direction (sink → host), it doesn't need the bidirectional shifter `Q3`/`Q4`/`Q5` use. `U8-19` (`HPD`, connector side) drives `Q6`'s gate directly; when asserted, `Q6` shorts the `INTERFACES_3V3`-via-`R69` pull-up node to `GND`, pulling `HDMI_HPD` low. **This means `HDMI_HPD` is active-LOW at RK3576 — the inverse of the connector's active-high `HPD` — so RK3576's HDMI driver/device-tree GPIO polarity needs to be configured for active-low or hotplug detection will read backwards.** Flagged prominently since it's a software/firmware dependency easy to lose track of once the schematic is done.
- **TMDS ESD protection now complete and clean for all 4 pairs, 2026-07-21 (final pass).** `D4`(Data1)/`D5`(Data2)/`D6`(Data0)/`D7`(Clock) — all `JLC_PRTR5V0U2X_C2827688`, a real 2-line 0.3pF low-capacitance TVS array, appropriately chosen for TMDS bandwidth — wired in-line between the connector and the RK3576-facing net, all following the same `HDMI_D#_PROC_8_*`/`HDMI_CLK_PROC_8_*` naming convention. The earlier stray `HDMI_D3`/`HDMI_D4` placeholder nets and the duplicate `D9` (a mistaken extra instance, not a needed one) are gone — user caught and fixed both. `standard-parts.md`'s discipline holds again (zero unresolved floating ESD-diode instances).
- **On the diff-pair voltage range you asked about**: RK3576's own HDMI PHY analog supplies are documented at 1.8V and 0.75V (`HDMI_TX_EDP_TX_AVDDIO1V8`, `AVDDCMN1V8`, `AVDDD0V75`, `AVDDC0V75` — see prior session's finding), not 3.3V — so the TMDS common-mode/swing is likely referenced well below the "classic 3.3V TMDS" assumption some ESD parts are sized around. `PRTR5V0U2X`'s ~5V working voltage / ~6V clamp (per its BOM description) won't false-trigger at a lower actual signal level — that's safe either way — but whether ~6V clamps *low enough* to protect RK3576's actual TMDS pad absolute-max rating isn't confirmed from anything ported so far (no pad-level absolute-max-voltage row found, only the supply-domain values). Same part is already used elsewhere on the board (`D1`), so it's a reasonable, standardization-consistent choice — just flagging the one number I can't independently confirm.

Not applicable in the usual sense for a passive connector — pin 18 (`+5V`) is a power line the **host supplies outward** to a connected sink (for the sink's own DDC EEPROM, typically current-limited), not a supply feeding the connector body. See Pin table.

## Known gaps (things the vendor datasheet does not specify)

- **No pin-function table anywhere in this datasheet** — the entire signal-assignment column above comes from the public HDMI Type-A standard, not this document. Recommend a second-source check (a known-good HDMI reference schematic) before trusting pin-for-pin, especially TMDS polarity — a +/- swap is a real, hard-to-debug signal-integrity failure, not a cosmetic error.
- **No differential impedance spec** (TMDS pairs are conventionally ~100 ohm differential) given anywhere in this datasheet — will need an external reference or the HDMI spec itself for controlled-impedance layout.
- ~~No ESD/surge rating — not addressed anywhere yet~~ Resolved 2026-07-21 — `D4`(Data1)/`D5`(Data2)/`D6`(Data0)/`D7`(Clock), all `JLC_PRTR5V0U2X_C2827688`, now protect all four differential pairs, cleanly named and with no stray/duplicate diodes left over.
- **RK3576's exact `D0`-`D3` → TMDS Data0/Data1/Data2/Clock lane mapping is unconfirmed** — needs the RK3576 HDMI TRM chapter (likely in a Part2 TRM volume not present in `cradle_sidecar/data/datasheets/`) or a Rockchip reference schematic.
- ~~Pin 14 (Reserved)~~ Resolved 2026-07-21, folded into the pinout correction — pin 14 is actually `CEC` on this connector (not Reserved/Utility, which turned out to be pin 17 instead). Left N.C. by user, a legitimate simplification for a display-only, non-ARC output.
- ~~`DDC_SCL`/`DDC_SDA`/`HPD` need level-shifting — not yet designed~~ Resolved 2026-07-21 for all four signals. `CEC`/`SCL`/`SDA` use the bidirectional `LBSS138LT1G`-per-line shifter (`Q3`/`Q4`/`Q5`). `HPD` (`Q6`) intentionally uses a simpler unidirectional inverting switch instead (confirmed by the user, not a wiring mistake — see High-level summary and Pin table row 19) — **this makes `HDMI_HPD` active-low at RK3576, which needs to be reflected in the HDMI driver/device-tree GPIO polarity config, not just the schematic.**
- **TMDS pad absolute-max voltage vs. the ESD diode's ~6V clamp not cross-checked** — RK3576's HDMI PHY analog supplies are 1.8V/0.75V-class (not 3.3V), so the actual signal swing is likely well below what a "5V-class" ESD part assumes; that's safe against false-triggering, but whether the diode's clamp voltage is low enough to protect the actual pad hasn't been confirmed against a specific RK3576 pad rating (none found in the ported tables).
- **Pull-up resistor values (4.7kOhm, `R62`-`R69`) look like a reasonable choice** for DDC-speed signaling (much lower than the eMMC bus's 30kOhm weak pulls, appropriately so — see prior session's reasoning) but haven't been checked against the specific VESA E-DDC/HDMI DDC electrical spec or a reference design for the target clock rate/cable capacitance.
- **Exact RK3576 GPIO bank for `SCL`/`SDA`/`HPDIN`/`CEC` still not confirmed from the ballout table directly** — the schematic's `INTERFACES_3V3` gate-bias choice is strong evidence the design *intends* a 3.3V-capable bank, but the actual Compute-SoC-sheet pinmux assignment (each signal has M0/M1 options on different balls) will be the real confirmation.
