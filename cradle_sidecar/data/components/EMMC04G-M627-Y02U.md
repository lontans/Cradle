## High-level summary

- Kingston-branded e-MMC 5.1 module, single-chip MMC controller + MLC NAND array, JEDEC standard package `[VERIFIED — datasheet p.6]`. FBGA-153, 14x14 ball grid with 43 depopulated positions (153 real balls) `[VERIFIED — datasheet p.41, Figure 3]`. 4GB density, ROHS, quantity 1 `[VERIFIED — Cradle.csv BOM row, designator U3]`.
- Host interface: standard e-MMC bus — CLK, CMD, DAT0-7 (8-bit data bus supported), RST_n (hardware reset), DS (data strobe, HS400 only) `[VERIFIED — datasheet p.7-8, Table 4]`.
- Vendor's own reference operating point for this exact part number (Table 1, Device Summary): **VCC = 3.3V, VCCQ = 1.8V or 3.3V (selectable)** `[VERIFIED — datasheet p.5, Table 1]`. The datasheet's own published performance/power numbers (Table 2/3) are measured at **VCC=3.3V, VCCQ=1.8V, HS400 mode, 8-bit bus, 200MHz DDR** — 250MB/s read / 25MB/s write sequential, 85.7mA/37.3mA read/write current, 0.13mA standby `[VERIFIED — datasheet p.5, Tables 2-3]`. Treat this as the vendor-intended mode, not just one option among many — see Power domains.
- **Correction, 2026-07-20**: earlier sessions treated `datasheet_quickstart.py`'s "no candidate integration pages" finding as proof this datasheet has no application/reference circuit at all — wrong. Figure 6 "Bus Circuitry Diagram" (p.21) is a real bus-level reference circuit; the auto-scan just missed it, a third auto-tool miss on this datasheet (see Known gaps). Decoupling values for VCC/VCCQ/VDDi, plus pull-up resistances for CMD/DAT0-7/RST_n, come from Table 11 "Signal Line Load" (p.26), not a schematic — see Pin table and Known gaps. Table 11 itself was garbled by the mechanical table-porter (merged OCR cells split "VDDi" into "V...DDi" across columns) and had to be re-read directly from the rendered page — worth remembering this specific table name/page if the auto-port ever needs re-checking.
- **Cradle is targeting HS400, not HS200, as the steady-state bus speed mode.** Reasoning: RK3576's dedicated "EMMC Host Controller" (TRM Chapter 38, distinct from the separate "Mobile Storage Host Controller (SDMMC&SDIO)" in Chapter 39 that handles SD/SDIO) has a Command Queuing Engine (38.3.6) — a feature that only exists to serve eMMC 5.1's command-queuing feature, i.e. the controller is built for the full eMMC 5.1 spec, not just legacy MMC `[VERIFIED — RK3576-TRM.outline.md, Chapter 38 section list]`. More directly: RK3576's own ballout physically allocates a dedicated `EMMC_STRB` pin (this card's `DS` ball, H5) — a signal with no purpose whatsoever outside HS400 — so the host hardware wiring itself is evidence HS400 is supported, not just theoretically possible `[VERIFIED — RK3576 datasheet V1.6.tables.md, EMMC_STRB pinmux row]`. This matches Kingston's own reference performance/power numbers (Table 2/3, p.5), which are measured at exactly HS400/8-bit/1.8V.
- **HS200 is not a separate mode Cradle "chooses" instead of HS400 — it's a mandatory intermediate step in HS400's own bring-up sequence.** Per the eMMC 5.1 spec, switching into HS400 requires the host to first put the device into HS200 and run tuning (CMD21) to characterize the read-data sampling window, then switch to HS400 `[MANUAL — user, 2026-07-20, standard JEDEC eMMC5.1 HS400 entry sequence, not found written out explicitly anywhere in this Kingston datasheet]`. So targeting HS400 means the host driver/BSP needs working HS200 tuning support regardless — there's no way to "skip" HS200 and land on HS400 directly.
- **If HS400/HS200 tuning turns out unreliable during RK3576 bring-up** (a real risk on a brand-new SoC's early BSP/driver support, independent of the eMMC device itself), the practical fallback is **High Speed DDR** (Table 6, p.8: 104MB/s, 8-bit, no tuning required, no `DS`/`EMMC_STRB` needed, `VCCQ` can stay 1.8V or 3.3V) — not HS200 alone, since HS200 needs the same tuning infrastructure as HS400 for half the reward. Dropping to HS DDR removes `DS` from the routing requirement entirely.
- **Storage sheet wired as of the 2026-07-20 22:26/22:31 netlist/BOM export.** `CMD`/`DAT0-7`/`RST_n` now sit on real hierarchical port nets named `EMMC_CMD`/`EMMC_D0`-`EMMC_D7`/`EMMC_RST_N` — matching RK3576's own `EMMC_*` ballout naming exactly, confirming these are meant to cross straight into the Compute-SoC sheet once it exists (`pending-port`, not orphaned). `CLK` similarly got `EMMC_CLK`. Each of `CMD`/`DAT0-7`/`RST_n` also picked up a 30kOhm pull-up to `STORAGE_1V8` (10 new `JLC_30K_CR0402FF3002G` resistors, `R51`-`R60`) — user standardized on the same 30k part already used for AP6275S's SDIO pull-ups rather than three different values per Kingston's per-line ranges; 30k sits comfortably inside all three (`R_CMD`/`R_RSTn` 4.7-50kOhm, `R_DAT` 10-50kOhm), so this is a legitimate simplification, not a compromise. `VCC` is wired to `STORAGE_3V3`, `VCCQ` to `STORAGE_1V8` (confirms the 1.8V/HS400 lean from the prior session as an actual wiring decision, not just a recommendation), `VDDi` has its own 2.2uF cap (`C99`) to `GND` (matches the earlier recommendation exactly), and all `VSS`/`VSSQ` balls confirmed on the single board `GND` net (no split ground needed, as reasoned earlier). (Caveat: the BOM groups these new resistors/caps into rows whose `SheetNumber` spans multiple sheets, so `netlist_parse.py` reports their sheet as "ambiguous" — that's just a BOM-row grouping artifact, the net membership itself is unambiguous.)
- **`DS` wired as a follow-up, same session (2026-07-20 22:48/22:49 export).** Now on net `EMMC_DS` with a 30kOhm pull-down to `GND` — a genuine pull-*down* (not pull-up like the rest of the bus), matching Figure 6 (p.21) and Table 23's `R_DS` spec (10-100kOhm, p.38). Every signal on this sheet is now wired; `EMMC_DS`'s resistor is the one component still shown as unannotated `R?` in the netlist export (confirmed becomes `R61` on the next re-export — cosmetic, not a wiring gap).
